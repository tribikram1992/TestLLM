14 ═══════════════════════════════════════════════════════════════════════
15 ▶ TECHNIQUE 1: EQUIVALENCE PARTITIONING (EP)
16 WHAT: Divide input data into classes where all values behave the same way.
17 APPLY WHEN: Any input has identifiable valid and invalid classes.
18 HOW:
19 For each input field, identify valid and invalid partitions.
20 Produce ONE test condition per partition (representative value).
21 Title format: "Verify [behavior] when [field] is [partition description]"
22 EXAMPLE (text field):
23 "Verify username is accepted when value contains valid alphanumeric chars"
24 "Verify username is rejected when value contains special characters"
25 EXAMPLE (file upload):
26 "Verify upload is accepted when file format is PDF"
27 "Verify upload is rejected when file format is EXE"
28 ▶ TECHNIQUE 2: BOUNDARY VALUE ANALYSIS (BVA)
29 WHAT: Test at the EXACT EDGES of input ranges. Defects hide at boundaries.
30 APPLY WHEN: Any input has limits — numeric, string length, date, file size, count.
31 HOW:
32 For each bounded field, produce conditions for: min-1, min, min+1,
33 max-1, max, max+1.
34 Title format: "Verify [behavior] when [field] is [exact boundary value]"
35 EXAMPLE (string length):
36 "Verify comment is rejected when length is 0 (just below min)"
37 "Verify comment is accepted when length is exactly 500 (at max)"
38 "Verify comment is rejected when length is 501 (just above max)"
39 EXAMPLE (date range):
40 "Verify booking is rejected when date is yesterday (past)"
41 "Verify booking is accepted when date is exactly today (at min)"
42 ▶ TECHNIQUE 3: DECISION TABLE
43 WHAT: Test combinations of conditions that lead to different outcomes.
44 APPLY WHEN: Multiple conditions/business rules combine to determine behavior.
45 HOW:
46 Identify all conditions involved.
47 Create rules for meaningful combinations — each producing a DIFFERENT outcome.
48 Title format: "Verify [outcome] when [condition1=value, condition2=value...]"
49 EXAMPLE (access control):
50 "Verify access granted when: logged_in=Y, has_admin_role=Y"

51 "Verify 401 when: logged_in=N"

53 ▶ TECHNIQUE 4: STATE TRANSITION
54 WHAT: Test movement between states in a workflow.
55 APPLY WHEN: The feature has a state machine, workflow, or lifecycle.
56 HOW:
57 Map all states.
58 Produce conditions for: valid transitions, invalid transitions, rollback paths.
59 Title format: "Verify state transitions from [A] to [B] when [trigger]"

61 "Verify state moves from Cart to Placed when user clicks Checkout"

65 APPLY WHEN: 3+ factors with multiple values each combine.
66 HOW:
67 Identify factors and their values.
68 Generate test conditions covering all PAIRS (not full Cartesian product).
69 Title format: "Verify [feature] works for [factor1=value, factor2=value, ...]"
70 EXAMPLE (compatibility):

72 "Verify feature works for: browser=Firefox, OS=macOS, role=Viewer"
