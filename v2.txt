═══════════════════════════════════════════════════════════════════════
THE 5 TECHNIQUES YOU MUST APPLY (IN ORDER)
═══════════════════════════════════════════════════════════════════════

▶ TECHNIQUE 1: EQUIVALENCE PARTITIONING (EP)
   WHAT: Divide input data into classes where all values behave the same way.
   APPLY WHEN: Any input field has identifiable valid and invalid ranges/classes.
   DETERMINISM:
   - For each input field, identify partitions in this order:
     1. Valid partition(s) — sorted by natural ordering
     2. Below-minimum partition
     3. Above-maximum partition
     4. Zero (if applicable)
     5. Negative (if numeric)
     6. Empty/null partition
     7. Wrong-type partition
   - Produce ONE test case per partition.
   - Within a field, order test cases by partition number above.
   
   HOW TO WRITE EXECUTABLE TEST CASES:
   - Use exact, concrete test data (not ranges: e.g., $25.50, not "$1-$50K")
   - Specify the EXACT UI element, input field, or API parameter
   - State the EXACT expected output (e.g., "Error message: 'Amount must be...'" 
     not "error occurs")
   - Include pre-conditions, steps, expected results

   EXAMPLE TEST CASE (NOT just a condition):
   - Test Name: "Transfer amount accepts valid value in middle of range"
   - Pre-condition: User logged in, has two accounts
   - Step 1: Navigate to Transfer page
   - Step 2: Enter amount: $2,500.00
   - Step 3: Click Submit
   - Expected Result: "System displays confirmation page with amount $2,500.00"
   - Evidence: Screenshot of confirmation page

▶ TECHNIQUE 2: BOUNDARY VALUE ANALYSIS (BVA)
   WHAT: Test at the EXACT EDGES of input ranges. Defects hide at boundaries.
   APPLY WHEN: Any input has numeric, date, or length constraints (min/max).
   DETERMINISM:
   - For each bounded field, produce test cases in this order:
     1. min-1 (just below minimum)
     2. min (at minimum boundary)
     3. min+1 (just above minimum)
     4. max-1 (just below maximum)
     5. max (at maximum boundary)
     6. max+1 (just above maximum)
   - Use exact calculated values (e.g., if min=$1.00, then min-1=$0.99, not "below min")
   - Order within field alphabetically; order across fields by field name.

   HOW TO WRITE EXECUTABLE TEST CASES:
   - Specify the EXACT boundary value in test data
   - State the exact expected behavior (pass/fail, error message, UI state)
   - Include all steps to input and verify

   EXAMPLE TEST CASE:
   - Test Name: "Transfer rejects amount exactly one cent below minimum ($0.99)"
   - Pre-condition: User logged in, transfer page open
   - Step 1: Enter $0.99 in Amount field
   - Step 2: Click Submit
   - Expected Result: "Error message displayed: 'Amount must be at least $1.00'"
   - Evidence: Screenshot showing error message

▶ TECHNIQUE 3: DECISION TABLE
   WHAT: Test combinations of conditions that lead to different outcomes.
   APPLY WHEN: Multiple conditions/business rules combine to determine behavior.
   DETERMINISM:
   - Identify ALL conditions from requirement (e.g., logged_in, valid_amount, 
     sufficient_balance, different_accounts, etc.)
   - Order conditions alphabetically by name
   - Assign each condition values (Y/N, T/F, or enumerated values)
   - Generate rules in this order:
     1. ALL conditions = positive result (success path)
     2. EACH condition = failure individually (one condition false at a time)
     3. COMBINATIONS producing distinct outcomes (ordered by condition position)
   - Eliminate duplicate outcomes (same result, different rule) — keep only one
   - For distinct outcomes, include all combinations
   - Within each rule type, order by rule ID (DT-Rule-001, etc.)

   HOW TO WRITE EXECUTABLE TEST CASES:
   - Test Name includes all condition values (e.g., "...when logged_in=Y, 
     valid_amount=Y, sufficient_balance=Y, different_accounts=Y")
   - Each condition maps to a concrete step (login check, amount validation, etc.)
   - Expected result is the SPECIFIC outcome for that rule
   - Include steps to set up each condition and execute the transaction

   EXAMPLE TEST CASES:
   - Test Name: "Transfer succeeds when logged_in=Y, valid_amount=Y, 
     sufficient_balance=Y, different_accounts=Y"
   - Expected Result: "Transfer completed, confirmation displayed"
   
   - Test Name: "Transfer rejected when logged_in=N"
   - Expected Result: "401 Unauthorized error displayed"
   
   - Test Name: "Transfer rejected when sufficient_balance=N"
   - Expected Result: "Error message: 'Insufficient funds available'"

▶ TECHNIQUE 4: STATE TRANSITION
   WHAT: Test movement between states in a workflow.
   APPLY WHEN: The feature has a state machine, workflow, or lifecycle.
   DETERMINISM:
   - Map all states from the requirement (e.g., Initiated, Validating, Debiting, 
     Crediting, Completed, Rolled Back, Failed)
   - Order states by execution sequence or alphabetically if no sequence
   - For each state, generate test cases in this order:
     1. Valid transition FROM this state (forward, normal path)
     2. EACH invalid transition FROM this state (error recovery)
     3. Rollback/error recovery paths
   - Order by source state, then by destination state

   HOW TO WRITE EXECUTABLE TEST CASES:
   - Test Name: "Verify state moves from [State A] to [State B] when [trigger]"
   - Pre-condition: System in State A (specify how to reach it)
   - Steps: Execute trigger action, monitor state change
   - Expected Result: "System state is now [State B]"
   - Evidence: Log entry, UI indicator, or API response showing new state

   EXAMPLE TEST CASES:
   - Test Name: "Verify transfer moves from Initiated to Validating when 
     Submit clicked"
   - Expected Result: "Backend logs show state = Validating; UI shows spinner"
   
   - Test Name: "Verify transfer moves from Debiting to Rollback when credit 
     fails"
   - Expected Result: "Backend logs show Rollback initiated; debit reversal 
     processed; state = Rolled Back"

▶ TECHNIQUE 5: PAIRWISE COMBINATION
   WHAT: Cover all PAIRS of multi-factor combinations efficiently.
   APPLY WHEN: 3+ factors with multiple values each combine (e.g., account types 
   × currencies × channels).
   DETERMINISM:
   - Identify all factors: list factor names alphabetically
   - For each factor, list values in a consistent order (alpha, or enum order)
   - Generate minimal pairwise cover (each pair of values covered at least once)
   - Order test cases by: first factor value (A-Z), then second factor, etc.
   - Use standard pairwise generation algorithm (e.g., PICT-equivalent)

   HOW TO WRITE EXECUTABLE TEST CASES:
   - Test Name: "Verify transfer works for account_type=[value], 
     currency=[value], channel=[value]"
   - Pre-condition: User set up with specified account and currency
   - Steps: Execute transfer via specified channel
   - Expected Result: "Transfer succeeds; confirmation in specified currency"
   - Evidence: Screenshot or transaction log

   EXAMPLE TEST CASES:
   - Test Name: "Verify transfer works for account_type=Checking, 
     currency=USD, channel=Web"
   - Expected Result: "Transfer completed; USD amount and Checking account shown"
   
   - Test Name: "Verify transfer works for account_type=Savings, 
     currency=EUR, channel=Mobile"
   - Expected Result: "Transfer completed; EUR amount and Savings account shown"

TECHNIQUE COVERAGE RULES (STRICT)
═══════════════════════════════════════════════════════════════════════

1. APPLY ALL 5 TECHNIQUES if the requirement supports them.
2. Skip ONLY if genuinely not applicable:

   - State Transition: 2-5 test cases covering main path + error paths
   - Pairwise Combination: 3-10 test cases covering all pairs

   Quality and coverage matter more than quantity.

5. EACH TEST CASE IS ATOMIC:
   - One primary behavior verified

100% REQUIREMENT COVERAGE ALGORITHM
═══════════════════════════════════════════════════════════════════════

1. PARSE REQUIREMENT into atomic clauses (sentences and sub-clauses).
2. FOR EACH clause:
   a. Identify inputs, constraints, conditions, and expected outputs
   b. Map to one or more applicable techniques
   c. Generate test case(s) that directly verify that clause
3. BUILD COVERAGE MATRIX mapping each clause to test case IDs
4. VALIDATE: Every clause has at least one test case. If not, add tests.
5. OUTPUT: Include the matrix showing 100% coverage

Example:
  Requirement: "System must transfer funds between two accounts of the same 
                customer. Amount must be between $1.00 and $50,000.00. 
                Transfer is rejected if insufficient balance."

  Clause 1: "Transfer funds between two accounts"
    → Test: DT-based test with same_customer=Y
    → Test: DT-based test with same_customer=N (negative)
  
  Clause 2: "Amount must be between $1.00 and $50,000.00"
    → Test: BVA tests for $0.99, $1.00, $1.01, $49,999.99, $50,000.00, $50,000.01
    → Test: EP test for valid range ($2,500.00)
    → Test: EP test for below-minimum
    → Test: EP test for above-maximum
  
  Clause 3: "Transfer is rejected if insufficient balance"
    → Test: DT-based test with sufficient_balance=N
    → Test: Negative test with specific balance amount
